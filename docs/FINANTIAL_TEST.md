# FinMate가 반드시 지켜야 할 업무 규칙과 금융 불변식을 테스트 및 검증
# 이를 통해 github에 push할때마다 github Actions를 통해서 자동으로 테스트 및 검증을 수행하고 결과를 조회하는 ci를 구축한다.
# 뿐만 아니라 에이전트가 요청과 목표에따라 기능을 구현할때마다 테스트가 추가되며, 해당 테스트들을 기반으로 시스템을 테스트한다.

# 1. 로그인
1-1. 로그인하지 않은 사용자는 기본 홈페이지, 로그인페이지, 회원가입 페이지말고 다른페이지에 접근할 수 없다.
1-2. 한번 로그인한 사용자는 세션과 쿠키 기반으로 매번 새로 로그인할 필요없이 세션과 쿠키 기반 검증을 통해 로그인을 수행한다.

# 2. 계좌
2-1. 한사람당 최대 10개의 계좌를 개설할 수 있다. 그 이상 개설시 예외가 발생한다.
2-2. 계좌이체는 트랜잭션과 락을 걸어서 여러 스레드에서 동시에 값(금액)을 수정하지 못하도록 해야한다(정합성을 보장해야 한다) -> 멀티스레드를 통해서 값 검증
2-3. 계좌이체를 할때에는 한번 이체할때 해당 계좌에 적용되어 있는 일회 이체한도를 넘길 수 없다.
2-4. 계좌이체를 할때에는 한번 이체할때 해당 계좌에 적용되어 있는 (일일 이체한도 - 사용자가 금일 이체한 금액)을 넘길 수 없다.
2-5. 계좌이체를 하기 위해 두 계좌에 lock을 획득할 때에는 데드락을 방지하기 위해 계좌번호가 작은 계좌부터 lock을 획득해야 하며, 데드락이 발생하지 않는지를 테스트한다.
2-6. 계좌이체를 성공하고, 커밋을 수행하기 전에, 계좌 이체 로그를 먼저 우선적으로 기록하고, 계좌 이체 로그 작성에 실패할 경우, 계좌이체도 롤백한다.
2-7. 계좌 개설시에는 고유한 계좌번호를 생성해서 이를 계좌번호 엔티티에 일치하는 계좌번호가 있는지를 먼저 검사하고, 이후에 계좌를 개설한다. 이때 데이터베이스에도 unique 제약조건으로 동일한 계좌번호가 생성되는것을 원천적으로 방지해야 한다.
2-8. 사용자는 본인 명의의 계좌를 제외하고, 다른 사람의 계좌를 볼 수 없다.

# 3. 증권
3-1. 한사람당 최대 10개의 증권 계좌를 개설할 수 있다. 그 이상 개설시 예외가 발생한다.
3-2. 증권계좌 <-> 일반계좌로 예수금 입출금의 경우에도 계좌와 마찬가지로 이체한도 제약을 적용받으며, 계좌와 마찬가지로 데드락을 방지하기 위해 항상 일반 계좌 먼저 lock을 획득한다.
3-3. 예수금 입출금의 경우에도 커밋을 수행하기 전에 이체 로그를 먼저 우선적으로 기록하고, 이체 로그 작성에 실패할 경우 , 예수금 입출금도 롤백한다.
3-4. 예수금 입출금의 경우에도 고유한 계좌번호를 생성해서 이를 계좌번호 엔티티에 일치하는 계좌번호가 있는지를 먼저 검사하고, 이후에 계좌를 개설한다. 이때 이는 동기화를 통해 동시에 동일한 계좌번호가 생성되지 않아야 한다.
3-5. 증권 계좌는 원화와 달러를 통화로 가져야 하며, 각 통화를 구분해서 관리하며, 두 통화가 섞이지 않아야 한다.
3-6. 사용자는 본인 명의의 증권 계좌를 제외하고, 다른 사람의 증권 계좌를 볼 수 없다.
3-7. 증권계좌의 경우에는 종목 거래를 위해 예약된 금액 및 종목들이 있기 때문에, LockedAmount와 AvailableAmount가 나눠져 있으며, 종목 거래나 예수금 입출금의 경우에는 AvailableAmount 기준으로 수행되어야 하며, TotalAmount기준으로, 즉 AvailableAmount를 초과하는 금액이나 종목만큼 거래하면 안된다.

# 4. 환전
4-1. 환전은 가장 최근 환율을 기반으로 수행하며, 환전을 위해 증권 계좌의 통화 (원화 / 달러)에 lock을 걸때에도 데드락을 방지하기 위해 항상 원화 먼저 lock을 획득한다
4-2. 환전의 경우에도 커밋을 수행하기 전에 환전 로그를 먼저 우선적으로 기록하고, 환전 로그 작성에 실패할 경우, 환전도 롤백한다.

# 5. 종목 구매
5-1. 종목 구메의 경우 국내 종목의 경우에는 각 종목 시장의 장 오픈 시간 + 애프터마켓 타임 사이에만 종목 거래가 가능하며, 장 마감시에는 거래가 불가능해야 한다.
5-2. 주문 체결의 경우에도 커밋을 하기전에, 로그를 먼저 저장하고 만약 로그 저장에 실패한다면 주문 체결도 롤백한다.
5-3. 시장가 주문이 아니라 지정가 주문 or 예약 주문의 경우에는 바로 주문이 체결되지 않을 수 있기 때문에, 해당 주문의 금액이나 종목 수량만큼을 lock을 해서 거래하지 못하게 해야한다. 만약 주문이 취소된다면 lock을 풀어 거래가 가능하게 한다.
5-4. 종목 거래도 현재 증권계좌의 AvailableAmount를 기준으로 거래가 가능해야 하며, 이를 초과하는 금액이나 수량을 기반으로는 주문할 수 없다,
5-5. 지정가 주문 및 예약 주문의 경우에는 반드시 주문 마감 기한을 설정해야 하며, 현재 날짜 및 시간보다 과거를 마감 기한으로 설정할 수 없어야 한다.
5-6. 종목 거래의 경우에도 데이터 정합성을 보장하기 위해 계좌 보유 금액이나 종목 보유 수량에 비관적 락을 걸어 동시성을 보장해야 한다.
5-7. 종목 구매의 경우 국내 종목은 증권 계좌의 원화 통화를 기반으로 거래해야 하며, 해외 종목(나스닥 종목)의 경우에는 증권 계좌의 달러 통화를 기반으로 거래해야 한다. 해외 종목을 원화로 거래하거나 국내 종목을 달러로 거래하면 안된다.
