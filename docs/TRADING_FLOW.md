# 주식 거래 흐름

이 문서는 현재 모의 주문의 접수·예약·체결·취소 순서를 설명한다. 예수금·보유 수량 보존, 원장 원자성, 체결·취소·만료 경합의 정합성 계약은 [금융 불변식](FINANCIAL_INVARIANTS.md)을 우선한다.

## 1. 공통 접수 준비

`OrderController`가 세션 사용자 ID와 요청 DTO를 `StockTradingCommandService`에 전달한다.

1. 종목과 본인 투자 계좌를 조회한다.
2. 투자 계좌는 `PESSIMISTIC_WRITE`로 잠근다.
3. 종목이 활성·거래 가능·거래 정지 아님을 검증한다.
4. 종목 시장이 정규장 또는 장후 시간외 거래 시간인지 검증한다.
5. 통화, 수량, 주문 유형, 가격과 만료 시각을 정규화한다.
6. 매수면 예수금, 매도면 보유 수량을 비관적 락 조회 후 예약한다.

## 1.1. 주문 가능 시간

일반 주문과 예약 주문 접수는 종목 시장별 정규장 또는 장후 시간외 거래 시간에만 허용한다.
실시간 payload로 기존 활성 주문·예약을 처리할 때도 같은 시간 정책을 적용하여 장외 stale price로
체결되거나 예약이 일반 주문으로 전환되지 않게 한다. 단, 만료 처리는 장외에도 수행할 수 있다.
주문 상세·주문 폼·포트폴리오 화면의 주문 진입/접수 버튼도 같은 시장별 시간 정책으로 비활성화하지만,
최종 권한 검증은 항상 `StockTradingCommandService`의 서버 검증이 담당한다.

| 시장 | 현지 기준 | 대한민국 시간 기준 |
|---|---|---|
| KOSPI | 대한민국 09:00~15:30, 15:40~18:00 | 09:00~15:30, 15:40~18:00 |
| KOSDAQ | 대한민국 09:00~15:30, 15:40~18:00 | 09:00~15:30, 15:40~18:00 |
| NASDAQ | America/New_York 09:30~16:00, 16:00~20:00 | 미국 서머타임 기간 22:30~05:00(+1일), 05:00~09:00(+1일). 그 외 23:30~06:00(+1일), 06:00~10:00(+1일) |

현재 정책은 주말만 제외한다. 한국·미국 공휴일, 조기폐장, 종목별 특수 거래 제한 캘린더는 **현재 구현되지 않음**.

## 2. 시장가 주문

### 매수

1. `StockTradingRealtimePriceService`가 최신 체결가 또는 호가에서 매수 실행 기준 가격을 얻는다.
2. `gross = 기준가격 × 수량`을 통화 단위로 올림한다.
3. 시장별 매수 수수료를 더한 금액을 `InvestmentCashBalance.lock()`으로 잠근다.
4. `StockOrder`를 `SUBMITTED`로 저장한다.
5. 실행 가격이 있으면 시장가 조건은 항상 참이므로 즉시 전량 체결한다.
6. 실행 가격이 없으면 활성 주문 이벤트를 발행하여 커밋 후 실시간 구독을 시작한다.

### 매도

1. `StockHolding.lockQuantity()`로 주문 수량을 잠근다.
2. 주문을 저장하고 실행 가격이 있으면 즉시 전량 체결한다.
3. 실행 가격이 없으면 KIS 실시간 체결가·호가 구독을 시작한다.

시장가 주문은 `expiresAt`을 저장하지 않는다. 실행 가격이 끝내 들어오지 않을 때 자동 만료시키는 별도 스케줄은 **현재 구현되지 않음**.

## 3. 지정가 주문

1. 매수는 `지정가 × 수량 + 수수료`를 잠근다. 매도는 수량을 잠근다.
2. 실시간 실행 기준 가격을 읽는다.
3. 지정가 매수는 실행 가격 `<= 주문 가격`, 지정가 매도는 실행 가격 `>= 주문 가격`일 때 체결한다.
4. 즉시 체결되지 않으면 커밋 후 `ACTIVE_ORDER` 목적의 KIS 실시간 구독을 추가한다.
5. payload 수신 때마다 해당 종목 활성 주문을 생성 시각 순서로 비관적 락 조회한다.
6. 만료 주문은 잔여 자산을 해제하고 `EXPIRED`로 변경한다.
7. 조건 충족 주문은 현재 남은 수량 전체를 체결한다.

## 4. 예약 주문

1. 일반 주문과 같은 방식으로 자산을 미리 잠근다.
2. `triggerCondition`, `triggerPrice`, 실제 주문 유형·가격을 가진 `StockOrderReservation(ACTIVE)`을 저장한다.
3. 커밋 후 `ACTIVE_RESERVATION` 목적의 실시간 구독을 시작한다.
4. payload 수신 시 해당 종목의 활성 예약을 생성 시각 순서로 비관적 락 조회한다.
5. 만료되면 자산을 풀고 `EXPIRED`로 변경한다.
6. 트리거 조건을 만족하면 `TRIGGERED`로 바꾸고 같은 자산 예약 값을 가진 `StockOrder`를 생성한다.
7. 새 주문이 즉시 체결되지 않으면 일반 활성 주문 구독을 추가한다.

트리거 조건의 구체 비교는 `StockOrderTriggerCondition.isSatisfied()`가 담당한다.

## 5. 실시간 주문 체결

```text
KIS payload
  -> KisRealtimeWebSocketClient
  -> KisRealtimeStore 최신값 저장
  -> KisRealtimePayloadReceivedEvent
  -> StockTradingRealtimeExecutionListener
  -> StockTradingExecutionService.processRealtimeUpdate(stockId)
       1) 예약 처리
       2) 일반 주문 처리
       3) 자산·상태·체결 원장 갱신
```

`processRealtimeUpdate()`는 `@Transactional`이다. 이벤트 리스너가 예외를 잡아 경고 로그를 남기므로 한 payload 처리 실패가 WebSocket 수신 루프를 직접 종료시키지는 않는다.

## 6. 예수금 예약 및 정산

아래 계산과 분기는 현재 구현의 운영 흐름이다. 이를 변경할 때 적용할 수치·반올림·보존식과 검증 기준은 [금융 불변식](FINANCIAL_INVARIANTS.md)을 따른다.

### 예약

- 지정가 매수: 지정가 기준 총액 + 수수료
- 시장가 매수: 현재 실행 기준 가격 기준 총액 + 수수료
- 시장가 예약 매수: 코드상 `triggerPrice`를 예약 기준으로 사용

### 체결

```text
grossAmount = 체결가 × 체결수량
commission = grossAmount × 시장별 수수료율
tax = 매도일 때 grossAmount × 시장별 세율, 매수는 0
매수 netCashAmount = grossAmount + commission
매도 netCashAmount = grossAmount - commission - tax
```

매수는 `settleBuyFromLocked(reservedCashAmount, netCashAmount)`로 정산한다.

- 잠금액 > 실제 정산액: 차액을 사용 가능 예수금으로 반환
- 잠금액 < 실제 정산액: 사용 가능 예수금에서 부족분 추가 차감
- 추가 예수금도 부족하면 트랜잭션 실패

매도는 보유 수량을 차감·잠금 해제한 뒤 `netCashAmount`를 사용 가능 예수금에 입금한다.

## 7. 보유 수량 변경

### 매수

```text
새 평균가 = (기존수량 × 기존평균가 + 체결수량 × 체결가) / 새 총수량
```

평균가는 소수점 6자리 `HALF_UP`이다. 수수료는 평균 매수가 계산에 포함되지 않는다.

### 매도

- `lockedQuantity`와 총 `quantity`를 체결 수량만큼 감소
- 잔여 수량이 0이면 평균 매수가를 0으로 초기화
- 실현 손익을 별도 엔티티나 필드로 계산·저장하는 로직은 **현재 구현되지 않음**.

## 8. 수수료와 세금

| 시장 | 수수료율 | 매도 세율 |
|---|---:|---:|
| KOSPI | 0.00015 | 0.0020 |
| KOSDAQ | 0.00015 | 0.0020 |
| NASDAQ | 0.0025 | 0.0000206 |

금액은 통화 소수 자릿수로 `HALF_UP`한다. 이 값이 실제 증권사 최신 요율과 일치하는지는 소스만으로 검증할 수 없으므로 **확인 필요**.

## 9. 취소와 구독 종료

- 활성 일반 주문 취소: 남은 예약 자산 해제 → 잔여 수량 취소 → 커밋 후 구독 감소
- 활성 예약 취소: 예약 자산 해제 → 상태 취소 → 커밋 후 구독 감소
- 구독 총 참조 수가 0이 되면 기본 60초 뒤 실제 KIS 구독 해제

부분 체결이 실제로 발생할 경우 `reservedCashAmount`와 `reservedStockQuantity`를 체결분만큼 줄이는 코드가 없다. 현재 실행 로직은 전량 체결만 수행하므로 노출되지 않지만, 부분 체결 도입 전 보완이 필요하다. **현재 구현되지 않음**.
